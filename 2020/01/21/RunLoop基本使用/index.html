应用能够持续运行，并且及时处理各种事件，全归功于RunLoop的存在，这些都是系统层面的，在了解了RunLoop基本介绍后，开发过程中可以利用RunLoop来干点什么呢？

一、定时器工作不受滑动事件影响。

如果不做处理，常常会发现，当UIScrollView滑动的时候，定时器不工作了，最常见的就是banner图的滚动，当滑动页面时banner不自动轮播了，因为当UIScrollView滑动时，当前RunLoop的Mode切换成了Tracking模式，解决这种问题的一种方案就是让自动轮播的定时器添加到当前RunLoop的Common模式下。

二、图片延迟显示减缓tableView滑动卡顿。

当UITableViewCell上有大图需要加载，滑动tableView时可能造成卡顿现象。可以考虑让加载图片的代码延迟在滑动事件结束后执行。

三、常驻线程（线程不死）

1、强引用指向线程（可行？）

正常来说，线程任务执行完后线程就会自动销毁，无法再使用。那么有人就会尝试用一个强引用把线程拿住，这样是否可行呢？

测试1打印结果：

MyThread dealloc

测试2打印结果：
— run
再次点击，程序崩了。。。

测试3打印结果：
— run
再次点击无打印输出，程序未崩溃

测试1：说明线程任务执行完后，线程销毁了；

测试2：说明无法通过强引用拿住线程，重复使用，线程在执行完没有执行dealloc，说明内存还在，但是线程已经处于消亡的状态，无法再次使用的，线程不能重复调用start方法；

测试3：首先要启动线程，即调用start方法，线程才能执行任务，在点击后只执行了一次run任务，后续再点击无输出，说明这种方式下线程同样处于消亡状态，无法执行任务了；

3、开启线程的RunLoop（正解）
1、RunLoop开启（添加source）

RunLoop内部是一个do-while循环，前面说过每个线程都有一个RunLoop与之对应，只有主线程的RunLoop是主动开启的，其他线程需要手动获取并开启，在线程任务中开启该线程的RunLoop，这个线程就一直处于一个do-while循环中，实现不死。

竟然打印了“RunLoop退出了”，正常来说RunLoop内部是个循环，不会走到这个打印的，现在打印了说明RunLoop退出了，进而线程也消亡从而达不到常驻线程的效果。RunLoop为什么会退出呢？

注意：RunLoop启动时(调用run)会先判断当前Mode是否为空（即Mode中Source，Timer，Observer三者一个都没有），如果为空则直接退出。

点击3次打印结果：
— run — <MyThread: 0x6000009b7680>{number = 5, name = (null)}
— run — <MyThread: 0x6000009b7680>{number = 5, name = (null)}
— run — <MyThread: 0x6000009b7680>{number = 5, name = (null)}

给RunLoop添加一个source，修改后可以看到，RunLoop没有退出，线程也没有消亡，实现了线程不死，还能不断接收处理事件。

    修改后RunLoop中已有timer，所有不会退出，timer开始工作，线程也实现了常驻，并且随时准备接收处理事件。

  • scheduledTimerWithTimeInterval方式

    前面说过scheduledTimerWithTimeInterval方法会自动把timer添加到当前线程的RunLoop中，这个方法相当于已经给线程创建了RunLoop，并且也将RunLoop中也有了timer，那么既然都欧有了，timer会工作吗？线程能常驻吗？

    测试后发现，并没有timer并没有工作，线程也没能达到常驻效果，为什么？RunLoop没有启动，其后执行[[NSRunLoop currentRunLoop] run];就好。

3、while+RunLoop

打印结果:
— run — <MyThread: 0x600001954740>{number = 6, name = (null)}
runloop退出了
…(无限打印)

再次点击
“runloop退出了”停止打印，输出：
— run — <MyThread: 0x600001954740>{number = 6, name = (null)}

当第一次点击时，执行run，进入while(1)死循环，RunLoop不断的开启后退出，因为它的Mode是空的，再次点击，即调用[self performSelector:@selector(run) onThread:self.thread withObject:nil waitUntilDone:NO];方法，performSelector相当于自动给RunLoop添加了Source，此时RunLoop的Mode不为空，RunLoop开启后就不会退出（内部do-while循环），这样也实现了线程不死和RunLoop的开启。

四、自动释放池

1、有什么用？

将一些调用autorelease的对象放到自动释放池中，当自动释放池被释放的时候，其中的所有对象都调用一次release方法。

2、自动释放池什么时候释放？

APP启动，主线程开始执行任务，默认开启它的RunLoop，RunLoop注册两个观察者。

观察者1监听RunLoop的kCFRunLoopEntry（即将进入RunLoop）状态，此时创建自动释放池。

观察者2监听RunLoop的kCFRunLoopBeforeWaiting（即将进入休眠）状态，此时释放旧的自动释放池，然后创建新的自动释放池，另外还监听RunLoop的kCFRunLoopExit（即将退出RunLoop）状态，此时释放自动释放池。

3、使用自动释放池

一般在开启RunLoop的时候可以使用自动释放池，让RunLoop里面的一些东西可以及时释放。注意看main函数开启RunLoop的时候也是用@autorelease包起来的。

五、总结使用场景

1、开启一个常驻线程

让一个子线程不死处于消亡状态，等待其他线程来发消息，处理其他事件。

(1)、在子线程开启定时器

​ 子线程开启定时器，避免和主线程的RunLoop的模式切换影响到定时器工作。

(2)、在子线程长期监控一些数据变化
2、控制定时器在特定模式下执行
3、让某些任务在特定模式下执行
4、添加Observer监听RunLoop的状态

比如监听卡顿，所有按钮点击事件之前等。
