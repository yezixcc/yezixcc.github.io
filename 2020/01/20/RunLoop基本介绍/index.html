都知道iOS程序中都有且仅有一条主线程，并且主线程不死，才能有应用的持续运行，为什么呢？这就和iOS中的RunLoop有关系了。

什么是RunLoop？
  • 1.Run：运行，Loop：循环，意思就是不停的跑圈；
  • 2.是一个对象，内部是一个do-while循环；
  • 3.基本作用：
    • (1) 保持程序的持续运行，让线程保活；
    • (2) 不断检测输入源和定时源来等待接受事件，通知线程处理事件；
    • (3) 处理App的各种事件（比如触摸，定时器，PerformSelector等事件，GCD async main queue）；
    • (4) 节省CPU资源，提高程序性能，没有事件处理的时候让线程休眠；

1、如果没有RunLoop会怎么样？—- 程序结束就死掉了。

2、有RunLoop就类似于以下情况（do-while），程序在持续的运行，不会结束。

RunLoop对象

iOS有两套API使用RunLoop，一个是Foundation框架的NSRunLoop（OC语言），一个Core Foundation框架的CFRunLoopRef（C语言），都能代表RunLoop对象；
NSRunLoop是CFRunLoopRef的一层OC封装；

RunLoop和线程

特点：

  • 每个线程都有唯一一个RunLoop与之对应，本质用字典存储的，key是线程，value是RunLoop对象；
  • 主线程的RunLoop是自动开启的，子线程RunLoop在获取的时候创建（懒加载方式），并非传统的alloc init 方式；
  • 在线程结束的时候RunLoop销毁；
  • 只能在当前线程内操作当前线程的RunLoop对象，不能在当前线程内操作其他线程的RunLoop对象；
  • 如果想让线程不死，可以开启该线程的RunLoop；

2、CFRunLoopModeRef：

简介：

（1）RunLoop运行模式；

（2）一个RunLoop包含若干个Mode，每个Mode包含若干个Source/Timer/Observer；

（3）每次RunLoop启动时，只能指定其中一个Mode，就是currentMode；

（4）如果希望切换Mode，需要退出当前循环，再重新指定一个Mode进入；

（5）不同组（Mode）的Source/Timer/Observer，互不影响，例如不同Mode有不同的定时器，就可以通过切换Mode来切换定时器；

（6）如果Mode里没有任何Source0/Source1/Timer/Observer，RunLoop会立马退出；

（7）系统默认注册了5个Mode：

几种模式：

  • (1) UIInitializationRunLoopMode：
    程序启动最先进入的Mode，启动完成不再使用，切换到kCFRunLoopDefaultMode；

  • (2) GSEventReceiveRunLoopMode：
    接收系统事件Mode，比如绘图渲染事件等，开发者通常用不上；

  • (3) kCFRunLoopDefaultMode：
    默认Mode，通常主线程的RunLoop就是在这个Mode下运行；

  • (4) UITrackingRunLoopMode：
    界面跟踪Mode，用于追踪ScrollView触摸滑动，保证界面滑动时不受其他Mode影响，当scrollView滚动的时候，主线程的RunLoop Mode会自动从kCFRunLoopDefaultMode切换成UITrackingRunLoopMode，所以此时kCFRunLoopDefaultMode模式下的定时器等都不将5工作；

  • (5) kCFRunLoopCommonModes：
    是一个占位模式，不是一种真正的模式，包含kCFRunLoopDefaultMode和UITrackingRunLoopMode两种模式；

4、CFRunLoopSourceRef

简介：

（1）事件源，输入源；

（2）按照概念分类：

  • Port-Based Sources

    基于端口的事件源，其他线程的或者系统内核发的一些消息。

  • Custom Input Sources

    自定义输入源。

  • Cocoa Perform Selector Sources

    一些perform selector事件。

（3）按照函数调用栈分类：

  • Source0：非基于Port的，比如按钮点击事件，performselector事件等，
  • Source1：基于Port的，通过内核和其他线程通信，接收，分发系统事件。

看看一个按钮点击事件的调用栈：

可以看到调用栈中有Source0，说明点击事件属于Source0，虽然调用栈中没有看到Source1，但是它是用来接收分发事件的，硬件层面的触摸事件，首先包装成一个Event，这个Event先到Souce1，然后由Source1分发到Source0，所有最后由Source0处理。

Event –> Source1 – > Source0 –> UIApplication –> UIWindow –> UIControl –> View
