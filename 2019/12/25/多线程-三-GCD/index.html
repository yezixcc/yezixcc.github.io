GCD：Grand Central Dispatch，中枢调度器，纯C语言，提供非常强大的函数，非面向对象；
苹果公司为多核的并行运算提供的解决方案；
自动利用CPU多核；
自动管理线程的生命周期，比如创建线程，调用任务，销毁线程。
没有线程的概念，只有队列和任务。

队列和任务

  • 任务：执行什么操作；

  • 队列：用来存放任务的。

  • 步骤：

    • 定制任务；
    • 将任务添加到队列中，GCD会自动的将队列中的任务取出，放到对应的线程中执行，队列遵循先进先出的原则。

  • 同步（sync）：

    • 只能在当前线程执行任务；
    • 立即执行添加的任务。

  • 异步（async）：

    • 可以在新的线程中执行任务，具备开新线程的能力，但不是一定就会开新线程；
    • 等当前函数调用完毕后再执行队列中的任务。

  • 并行队列（concurrent）：

    • 可以让多个任务并行执行；
    • 并发功能只在异步下才有效。

  • 串行队列（serial）：

    • 一个任务完成后再执行下一个任务。

基本使用

1. 异步+并行队列。

打印结果：
asyncConcurrent——–end
2—–<NSThread: 0x6000006f62c0>{number = 4, name = (null)}
1—–<NSThread: 0x6000006cd280>{number = 6, name = (null)}
3—–<NSThread: 0x6000006cd300>{number = 7, name = (null)}

结论：

先执行asyncConcurrent--------end，说明先执行完当前函数，再执行队列中的任务。

可以开启新线程，任务1，2，3在不同的新的线程中同时执行，打印的结果1，2，3顺序是不固定的。

2. 异步+串行队列。

打印结果：
asyncSerial —– end
1—–<NSThread: 0x600003a9f300>{number = 4, name = (null)}
2—–<NSThread: 0x600003a9f300>{number = 4, name = (null)}
3—–<NSThread: 0x600003a9f300>{number = 4, name = (null)}

结论：会开启一个新线程，但任务在新的线程中一个接着一个按顺序执行的。

3. 同步+并行队列。

打印结果：
1—–<NSThread: 0x60000021a100>{number = 1, name = main}
2—–<NSThread: 0x60000021a100>{number = 1, name = main}
3—–<NSThread: 0x60000021a100>{number = 1, name = main}
syncConcurrent——–end

结论：不会开启新线程，任务在当前线程中一个接着一个按顺序执行。

4. 同步+串行队列。

打印结果：
1—–<NSThread: 0x600003075c00>{number = 1, name = main}
2—–<NSThread: 0x600003075c00>{number = 1, name = main}
3—–<NSThread: 0x600003075c00>{number = 1, name = main}
syncSerial —– end

结论：不会开启新的线程，任务在当前线程中一个接着一个按顺序执行的。

5. 异步+主队列。

打印结果：
asyncMain —– end
1—–<NSThread: 0x6000003ad280>{number = 1, name = main}
2—–<NSThread: 0x6000003ad280>{number = 1, name = main}
3—–<NSThread: 0x6000003ad280>{number = 1, name = main}

结论：不会开启新的线程，任务在主线程中一个接着一个按顺序执行的。主队列默认是一个串行队列。

6. 同步+主队列。

打印结果：
syncMain —– begin

分析：

主队列就是回到主线程执行任务，在主线程中，主队列同步执行任务，而主队列又是个串行队列导致主线程卡死。队列换成其他非主队列就行。

使用sync函数往当前串行队列中添加任务，会卡住当前的串行队列。

打印结果1：
syncMain —– begin

打印结果2：
syncMain —– begin
4—–<NSThread: 0x6000000c2280>{number = 7, name = (null)}

各队列执行效果

并行队列 手动创建的串行队列 主队列
同步（sync） 没有开启新线程；
串行执行任务；		 没有开启新线程；
串行执行任务		 没有开启新线程；
串行执行任务；
异步（async） 有开启新线程；
并行执行任务；		 有开启新线程；
串行执行任务；		 没有开启新线程；
串行执行任务；
同步执行 异步执行
串行队列 在当前线程上，挨个执行，阻塞当前线程 在其他线程上，挨个执行，不阻塞当前线程
并行队列 在当前线程上，挨个执行，阻塞当前线程 开很多线程，一起执行（GCD也会根据资源的情况控制并行的数量，如果任务很多也不会让所有的任务都同时执行）

能不能开线程不由是否是并行和串行队列决定，而是看是同步执行还是异步执行。GCD开多少线程由系统繁忙程度确定，如果任务很多，GCD会开启适当的子线程，不会让所有任务同时执行。

其他常用函数

1. 阻塞队列：dispatch_barrier_async。

姑且称它为栅栏函数吧，函数之前添加的队列任务先执行，然后执行barrier函数中添加到队列的任务，之后再执行后面添加到队列的任务，可以起到一个同步的作用。

打印结果：
barrier—- end
—-1—–<NSThread: 0x600001a56000>{number = 7, name = (null)}
—-2—–<NSThread: 0x600001a7c480>{number = 3, name = (null)}
—-barrier—–<NSThread: 0x600001a56000>{number = 7, name = (null)}
—-3—–<NSThread: 0x600001a56000>{number = 7, name = (null)}
—-4—–<NSThread: 0x600001a7c480>{number = 3, name = (null)}

执行完dispatch_barrier_async前面的任务，再执行dispatch_barrier_async中的任务，最后执行dispatch_barrier_async后的任务。dispatch_barrier_async相当于一个栅栏，把队列中的任务拦住了，只有等栅栏前面的任务完成后才能继续执行后面的任务。

注意：queue不能是全局并行队列，否则barrier功能失效。

2.延时执行：dispatch_after。

延时函数，我们可以使用它来进行延时调用处理。

延时2s执行，dispatch_get_main_queue()可以换乘其他队列，默认是主队列执行。

3. 一次性代码：dispatch_once。

姑且称它为一次性函数吧，block中的代码在整个程序运行中只会执行一次，常用作单例的实现。

block中的代码在整个程序运行过程中只会走一次，但注意不要和懒加载混淆，不要用在懒加载中。

4. 遍历：dispatch_apply。

遍历函数，类似于一个for循环。block中的任务会执行count次，如果传入的队列queue是并行队列，则任务会并行执行，全部执行完成后再返回执行后面的代码，是一个同步调用。

5. 队列组：dispatch_group_async。

将很多队列放到一个组里面，当组里面所有任务都完成的时候发出一个通知。

任务1，任务2并行执行，等任务组中的任务全部完成后再执行dispatch_group_notify中的任务。比如有个页面要请求几个接口，要所有接口请求完成才执行刷新UI的操作，就可以使用dispatch_group_async实现。

以上都存在一个dispatch_async_f的方式，主要区别就是任务是以block的方式还是以函数方式，下面两种方式效果一样，都是异步提交任务task到队列中。
